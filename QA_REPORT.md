## 一、已修复问题

### 1. 云端合并时丢失「仅本地存在的打卡」

- **位置**：`pages/shelf/index.js` → `mergeCoffeeLogsPreserveCutout`
- **问题**：只遍历 `cloudLogs` 的日期。若用户离线新增了某日打卡，该日期在云端不存在，合并结果里没有这个日期，写回本地后该日打卡被覆盖丢失。
- **修复**：合并时日期取「云端 ∪ 本地」；对仅本地有的日期直接保留本地数据；对同一天云端与本地条数不一致时，保留云端条数并合并 cutoutUrl，若本地条数更多则把多出的条目接在后面。
- **状态**：已修复。

## 二、核心流程走查结果（无 bug）

| 流程 | 结论 |
|------|------|
| 打卡上传照片 | `uploadImage` 支持 openid 为空时用 anonymous；失败有 toast。 |
| 打卡提交 / 编辑 | 先写本地再同步云端；编辑用 editIndex 覆盖原条，不重复新增。 |
| 删除打卡记录 | 先收集该条所有 `url`/`cutoutUrl`（仅 cloud://）→ 调 `deleteCloudFiles` → 再 splice 并同步；删除后历史页会 refresh，咖啡架 onShow 会 loadShelf，不会出现「已删记录的照片仍挂在架上」。 |
| 咖啡架从相册添加 | 上传 → 写入 shelfItems 并同步云端 → loadShelf → 后台自动抠图；id 迁移逻辑存在。 |
| 咖啡架 shelf 从架上移除 | 先收集该项 `url`/`cutoutUrl` → deleteCloudFiles → 再从 shelfItems 中 filter 并同步。 |
| 咖啡架 log 从架上移除 | 仅写入 shelfHiddenLogIds，不删 coffeeLogs、不删云文件；loadShelf 时按 hiddenLogIds 过滤，符合需求。 |
| 云端同步（coffeeLogs / shelfItems / shelfHiddenLogIds） | 三路拉取、合并后写回本地，updated 时 `loadShelf(true)` 不再触发自动抠图；合并后保留本地 cutoutUrl、保留本地独有日期/条目（已修复）。 |
| 图片本地缓存 | 仅 cloud:// 走缓存；LRU 80 条；持久化在 USER_DATA_PATH；cached-image 先展示 url 再异步换成本地路径。 |
| 消费足迹地图 | 使用 MapContext.moveToLocation 将地图中心移到当前定位；删除/移除不依赖地图数据。 |

## 三、风险与边界（可接受 / 建议）

1. **云端删除失败不阻塞用户**  
   `deleteCloudFiles` 失败只打 log，本地已删/已移除，用户提示正常。可能导致云存储里残留文件，可接受；若需可后续加「删除失败」toast 或重试。

2. **同一 fileID 被多处引用**  
   当前设计下：打卡照片与咖啡架「从相册添加」的 fileID 不同；咖啡架展示的「来自打卡」的图与打卡记录共用同一 fileID。删除打卡记录时会把该条下的云文件删掉，该条已从列表移除，架上也不再显示该条，无重复引用误删问题。

3. **图片缓存与云端删除不同步**  
   删除/移除后云文件被删，本地缓存文件仍可能存在（直到 LRU 淘汰）。因对应槽位已从列表移除，不会再用到该缓存，仅占一点空间，可接受。

4. **cached-image 异步更新**  
   若某槽位 url 在很短时间内被改两次（如先原图再抠图），异步回调可能后到先至。实际使用中一个槽位先显示原图再切抠图，顺序稳定，风险极低。

5. **云函数 / 云存储不可用**  
   getOpenId、setJson、deleteCloudFiles、segmentImage 等失败时均有 catch 或 fallback，不会导致白屏或未捕获异常。

## 四、建议的后续验证（真机/工具）

- 删除一条带多张图的打卡 → 日历与咖啡架均不再显示该条；云存储中对应 fileID 已删（可在控制台查）。
- 咖啡架移除一张「从相册添加」的图 → 架上看不到该图；云存储中该图及抠图 fileID 已删。
- 离线新增一条打卡 → 再联网进咖啡架触发同步 → 合并后该日打卡仍在，无丢失（验证 merge 修复）。
- 咖啡架多次进入同一批图 → 第二次起加载明显更快（验证本地缓存）。

## 五、总结

- **已修复**：云端合并时保留本地独有日期与多余条目，避免离线新增打卡被覆盖。
- **整体**：删除/移除与云端删文件、打卡与咖啡架与同步与缓存逻辑一致，未发现其他逻辑 bug。
- **风险**：均为边界或可接受（删除失败不阻塞、缓存不主动失效、异步竞态极低），可按当前实现上线，再结合真机按第四节做一次回归即可。
